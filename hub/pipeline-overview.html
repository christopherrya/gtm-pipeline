Building a two-system outbound pipeline from scratch

How we went from 15,000 raw contacts in a spreadsheet to a fully automated cold email pipeline with CRM sync, ICP scoring, A/B testing, and re-engagement loops — in a single build session.

The core question was simple: how do you get a real estate disclosure product in front of 15,000 agents without a sales team, a marketing budget, or a year to build infrastructure?

The answer turned out to be architectural. Instead of buying a marketing platform with built-in everything, we designed a two-system pipeline — Twenty CRM as the contact source of truth and Instantly as the sending engine — stitched together by scripts that Claude Code generates, maintains, and operates. The automation layer isn't a SaaS product. It's the AI itself.

03 — Decisions
Tradeoffs we made

Every architectural choice traded something. These are the four decisions that shaped the system, why we made them, and what we accepted losing.

Twenty CRM over SuiteCRM

We traded SuiteCRM's mature workflows and reporting for Twenty's clean API and flexible data model. The bet: Claude Code is the automation layer, so we don't need built-in workflow engines. We need an API that's fast to script against. Twenty's batch endpoints and custom objects made 15k-record operations trivial. The risk is that Twenty is pre-1.0 software — we pin versions and accept breaking changes as the cost of velocity.

Gained: API-first flexibility

Two systems, not three

We dropped Supabase from the outbound path entirely. One fewer system to sync means one fewer class of "data got out of sync" bugs. The cost: no SQL-queryable analytics on campaign performance, and no backup data store if Twenty goes down. We accepted this because speed to market mattered more than architectural completeness. Supabase re-enters the picture for Bucket 2 (inbound disclosure processing) — but outbound stays clean and simple.

Gained: Simpler mental model

Polling over webhooks

We poll Instantly every 30 minutes instead of receiving webhooks. This eliminated the need for a public endpoint, SSL certs, and webhook verification — the CRM runs on a Mac Mini behind Tailscale. The tradeoff is latency: a hot lead replies and we find out 30 minutes later. For mass cold outreach, this is acceptable. For Bucket 2's targeted outreach where individual replies matter, we'll add real-time webhooks.

Cost: 30-min reply latency

Self-hosted over cloud

Running Twenty on a Mac Mini means zero per-contact costs at any scale — 15k now, potentially 100k later. Full Postgres access for raw queries and bulk fixes. The risk is uptime: no redundancy, no failover, no automatic backups unless we set up pg_dump. We mitigate with Tailscale for reliable access and a daily backup cron. If uptime becomes critical, we migrate to a Hetzner VM.

Cost: Single point of failure
04 — Bucket 2
Product-led outreach

The second motion uses Discloser's own product as the conversion mechanism. Reach out as a buyer, process returned disclosures through the AI pipeline, then send findings back to the agent. They onboard to see the full analysis — already converted before signup.

01 — OUTBOUND

Buyer inquiry

Scrape active listings at major brokerages. Send a personal buyer inquiry via Instantly referencing the specific property. Each email gets a unique reply-to address for automatic correlation.

02 — PROCESS

Auto-ingest disclosures

Agent replies with PDFs. Resend webhook parses attachments, uploads to storage, triggers the existing OCR + AI pipeline. Document AI extracts findings, Voyage AI generates embeddings. Zero manual steps.

03 — CONVERT

Findings as CTA

Send top 2-3 critical findings as teasers. CTA links to a signup page with their property pre-loaded. Email pre-filled, value demonstrated before account creation. 3-8% end-to-end conversion target.
